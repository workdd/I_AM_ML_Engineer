## 한 줄 요약

조직 지식을 **"YAML 프론트매터가 붙은 마크다운 파일들의 디렉토리"** 로 표현해 AI 에이전트끼리 교환 가능하게 만드는 개방형 포맷. Google Cloud 데이터 클라우드 팀이 v0.1 공개.

## 등장 배경 — "올바른 정보의 부재"

파운데이션 모델이 아무리 좋아져도 현장 문제는 결국 **조직 지식이 흩어져 있다**는 것. 테이블 스키마, 비즈니스 지표, 장애 대응 런북이 메타데이터 카탈로그·위키·코드 주석에 제각각 갇혀 있어, 에이전트가 매번 맥락을 재구성해야 한다.

- 근본 원인: 모든 벤더가 자기 카탈로그·SDK·스키마를 독립 제공 → **이식 불가능**
- Karpathy의 "LLM Wiki" 패턴(Obsidian 볼트, `CLAUDE.md`/`AGENTS.md`, `index.md`/`log.md` 등)이 여러 이름으로 반복 등장 → 표준화 필요성

## 핵심 설계 — 복잡한 스킴도 SDK도 없다

지식을 세 가지 "그냥(just)"으로 표현:

1. **그냥 마크다운** — GitHub에서 렌더링, 모든 에디터에서 읽힘
2. **그냥 파일** — tarball·git·파일시스템으로 배포
3. **그냥 YAML 프론트매터** — 소수 필드만 정의

### 번들 / 개념 / 개념 ID

- **번들(bundle)**: 마크다운 파일들의 디렉토리 = 배포 단위
- **개념(concept)**: 각 `.md`가 나타내는 테이블·지표·API·플레이북
- **개념 ID**: 파일 경로에서 `.md` 제거 (`tables/users.md` → `tables/users`)

### 개념 문서 구조

```markdown
---
type: BigQuery Table          # 유일한 필수 필드
title: Customer Orders        # 권장
description: One row per...    # 권장
resource: https://...         # 권장 (실제 자산 URI)
tags: [sales, orders]         # 선택
timestamp: 2026-05-28T...     # 선택
---

# Schema
| Column | Type | Description |
...
# Examples
...
# Citations
[1] [Reference URL]
```

### 상호 링크 · 예약 파일명

- 링크: 절대(`/tables/customers.md`, 권장) 또는 상대(`./other.md`). **링크의 의미(부모/참조/조인)는 주변 산문이 전달**하고, 소비자는 깨진 링크를 관용해야 함(아직 안 쓰인 지식일 수 있음).
- `index.md`: 점진적 공개(progressive disclosure), 디렉토리 열거
- `log.md`: ISO 8601 날짜별 변경 이력
- 그 외 모든 `.md`는 개념 문서

## 세 가지 원칙

| 원칙 | 의미 |
|------|------|
| **최소한의 주관** | 모든 개념에 요구하는 건 `type` 필드 하나뿐. 나머지는 생산자 자유 |
| **생산자/소비자 독립성** | 포맷이 곧 계약. 양 끝 도구(손으로 쓴 번들 ↔ AI 소비, 파이프라인 ↔ 시각화)는 독립 교체 가능 |
| **플랫폼이 아니라 포맷** | 독점 계정·SDK 불필요. "가치는 누가 소유하느냐가 아니라 얼마나 많은 당사자가 그 언어를 말하느냐에서" 나옴 |

**적합성도 관대함** — 소비자는 다음을 거부하면 안 됨: 선택 필드 누락, 모르는 type 값, 모르는 추가 키, 깨진 링크, `index.md` 부재.

## MCP와의 관계 (보완적)

- **MCP**: 에이전트가 외부 시스템과 **어떻게 대화하는가**
- **OKF**: 그 컨텍스트 지식을 **어떤 포맷으로 표현하는가**
- → 경쟁이 아니라 보완

## 함께 공개된 레퍼런스 구현 (의도적 PoC)

1. **인리치먼트 에이전트**: BigQuery 데이터셋 순회 → OKF 문서 초안 작성 → LLM 패스로 보강
2. **정적 HTML 시각화 도구**: OKF 번들을 백엔드 없는 자기완결 파일로 변환
3. **샘플 번들 3종**: GA4 이커머스 / Stack Overflow / Bitcoin 공개 데이터셋

> 핵심 강조: 기여물은 **"도구가 아니라 포맷 그 자체"**. 도구는 시도 비용을 낮추는 PoC일 뿐, 특정 LLM·프레임워크 의존성 없음.

## 결론·시사점

- 목표: 지식 교환의 **공용어(lingua franca)** 가 되는 것
- v0.1은 완성형이 아니라 **출발점**. 마이너=하위호환 추가, 메이저=호환성 깨는 변경
- Google의 제안: 명세 읽기 → 생산자 작성 → 소비자(뷰어·검색·에이전트) 작성 → 레퍼런스 구현 시험 → 이슈/PR 기여

## 메모 — 내 작업과의 연결

흥미롭게도 이 레포(`I_AM_ML_Engineer`)의 `readings/`·`CLAUDE.md` 구조 자체가 OKF가 말하는 "LLM Wiki 패턴"의 비공식 사례다. 마크다운 + 프론트매터(출처·저자·태그 헤더) + 상호 링크로 지식을 쌓는 방식이 OKF v0.1과 거의 동형. 만약 이 노트들에 `type` 필드를 단 YAML 프론트매터를 붙이면 그대로 OKF 번들이 된다.

멀티 에이전트 작업 관점에서도 의미 있다 — 에이전트들이 공유하는 컨텍스트(스키마, 플레이북, 지표 정의)를 OKF 번들로 두면, 각 에이전트가 동일한 지식 표면을 참조하게 되어 앞서 고민한 "에이전트 간 컨텍스트 파편화·재구성 비용"을 줄일 수 있다.
